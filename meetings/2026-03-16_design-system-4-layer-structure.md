## Phase 1: 전문가별 분석

### Design Director

Atomic Design 5단계를 실무 4단계로 압축한 구조. 각 계층의 책임이 명확하고 겹치지 않음.

| 제안 계층 | Atomic Design | shadcn/ui | TDS 현재 |
|-----------|--------------|-----------|----------|
| Foundation | Atoms (Tokens) | CSS | Variables |
| Components | Atoms + Molecules | components/ui/ | Primitives + Composed |
| Blocks | Organisms + Templates | blocks/ (27개) | 없음 |
| Patterns | Pages (플로우) | 없음 | 없음 |

Components가 atom~molecule을 평탄화하지만, 이는 shadcn과 동일한 의도적 단순화. 바이브코딩 AI에게는 atom/molecule 구분이 불필요한 복잡도.

### Engineering Lead

위→아래(Patterns→Foundation) 구체화 구조가 AI 트리 탐색과 일치하여 바이브코딩 매핑 우수.

현재 TDS 갭: Foundation/Components는 충족, Blocks/Patterns는 없음. 그러나 Start Simple 원칙 적용 시 컴포넌트 마이그레이션 완료 전에 Blocks/Patterns 추가는 빈 레이어만 생성.

Patterns는 코드보다 문서(플로우차트, 가이드라인)에 가까움. Figma 컴포넌트로 만들 대상 아님.

## Phase 2: 종합 토론

### 합의점
- 4계층 구조 자체는 건전 — 역할 분리 명확
- shadcn blocks와 1:1 호환
- TDS에 즉시 전면 적용 불필요 — 점진 도입
- Primitives/Composed가 이미 Components 역할 수행 중

### 충돌점
- DD: Blocks 빈 페이지 예약 (의도 전달)
- EL: 빈 레이어 관리 부담, 나중에 분리 (10개+ 시점)

### 추천
> **옵션 B** — 현재 Foundation + Components(Primitives/Composed) 유지. Blocks는 섹션 수준 컴포넌트 3건+ 시 자연 도입. Patterns는 문서로만 관리.

**TDS 매핑:**
```
Foundation  = TDS Variables (Primitives + Theme)
Components  = TDS Primitives + Composed
Blocks      = (향후) 섹션 수준 조합 3건+ 시 도입
Patterns    = (문서) 프로토타입/플로우 가이드
```
