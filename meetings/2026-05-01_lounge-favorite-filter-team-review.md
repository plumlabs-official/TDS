## Phase 1: Expert Analyses

### Design Director

Using User-Centered Design and Design System criteria, the user edit is stronger than the previous grouped-surface experiment. The black `업데이트` chip and neutral `즐겨찾기` chip create a clean control model without adding another avatar badge. This avoids conflict with `Creator Badge` and unread red-dot indicators.

The main risk is semantic ambiguity: the current state reads as a **view switch/filter**, not as "these specific visible lounges are favorites." If that is the intended interaction, it is the right direction. If the requirement is item-level favorite identity inside a mixed update list, this pattern needs an additional selected/favorite state elsewhere.

### Product Leader

Using Problem Space vs Solution Space: the underlying user problem is "I want quick access to lounges I care about and know which of them has activity." A filter switch solves that more cleanly than marking every item. The current design should be framed as:

- `업데이트`: activity-first view across relevant lounges
- `즐겨찾기`: favorites-only view

This avoids inventing a third visual state on each card. The product decision to confirm is whether favorites are a separate filtered view or a pinned subset inside the update view.

### Engineering Lead

The horizontal list can map to a simple scroll row. The main implementation caution is keeping the row card dimensions consistent; current metadata shows the first card at 90x108, middle cards at 90x88, and last at 100x82, so component props or overrides should be normalized before handoff.

## Phase 2: Synthesis

### Agreements

- The user-edited chip/filter direction is better than wrapping cards in a group background.
- Favorite should not be represented with another avatar badge because `Creator Badge` and unread red-dot already occupy that semantic space.
- The design should choose one mental model: either filter view or item-level flag. The current design is clearly better as a filter view.

### Conflicts

- Design wants the current top chip pattern kept for simplicity.
- Product wants the interaction semantics explicitly defined: `즐겨찾기` is a separate favorites-only state, not a passive label.
- Engineering wants Profile Card dimensions normalized before implementation handoff.

### Trade-Offs

| Option | Pros | Cons | Support |
|---|---|---|---|
| Keep chip filter model | Clean, familiar, low visual noise, no badge conflict | Does not identify favorites inside mixed list | Design, Engineering |
| Add item-level favorite markers | Identifies each favorite immediately | Reintroduces badge/metaphor conflict | Weak support |
| Show both favorite and general groups | Good for comparison | More complex and heavier than needed | QA only if filter is rejected |

### Recommendation

Keep the user-edited `업데이트` / `즐겨찾기` chip pattern as the primary direction, but define it as a **view switch**. Add or keep a sibling state where `즐겨찾기` is selected and the row shows favorites only. Do not add favorite markers to individual avatars in the `업데이트` state.

Before implementation handoff, normalize the `Profile Card` row dimensions so the list is built from one repeatable card contract.
